# validation-lab Agent v1

## 文档目的

本文件用于固化 `validation-lab`（市场验证实验室）agent 的第一版设计方案。

它不是随手讨论记录，而是后续迭代、反馈、升级时的基线文档。
未来如果要对这个 agent 做能力扩展、职责调整、工具接入、隔离增强，优先基于本文件进行评审和修订。

---

## 1. Agent 名称

- **Agent ID**: `validation-lab`
- **显示名称**: 市场验证实验室
- **版本**: `v1`

---

## 2. 一句话定位

`validation-lab` 是一个独立、可控、可停、可替换的市场验证实验室 agent。

它的职责是在项目进入真实世界之前，承担低成本、多轮、结构化的市场验证与反馈闭环，帮助主线业务减少方向性错误。

---

## 3. 核心目标

先把这个 agent 设计成一个独立的验证实验室。

这样以后如果需要接入 MiroFish 或其他验证工具，也不会污染主 OpenClaw 主系统。

---

## 4. 设计原则

### 原则 1：职责隔离
该 agent 只负责：
- 市场验证
- 叙事测试
- 版本比较
- 仿真验证
- 反馈归纳
- 修正建议

它不负责：
- 主线项目执行
- 主 OpenClaw 运行
- 主业务调度
- 生产系统运维

### 原则 2：环境隔离
未来使用的任何重依赖，都不能直接进入主系统。

必须做到：
- 单独目录
- 单独依赖环境
- 单独日志
- 单独数据目录
- 单独 agent 身份
- 单独启动 / 停止控制

底线：
- 它坏了，主 agent 还能正常工作
- 它崩了，OpenClaw 主系统不能被拖垮

### 原则 3：工具可插拔
市场验证 agent ≠ MiroFish agent。

MiroFish 只是未来可能接入的一个模块，而不是这个 agent 的全部身份。

可扩展路径：
- 现在：人工验证 + 结构化比较
- 以后：接 MiroFish
- 再以后：接 Reddit / X / Landing Page / 调研反馈

### 原则 4：先定流程，再装工具
先把以下内容定下来：
- 职责
- 输入输出
- round 机制
- 隔离边界

然后再决定是否接入 MiroFish。

## 5. 顶层结构

建议未来整体结构为：

```text
主 OpenClaw 系统
├── main agent
├── execution-audit
├── retrospective-loop
└── validation-lab agent
```

其中 `validation-lab` 的定位是：

> 所有“真实世界之前验证动作”的实验室。

它可以接收：

- 新市场假设

并输出：
- 风险点
- 理解偏差
- 版本比较
- 修正建议
- 是否值得进入真实世界

## 6. 目录隔离方案

建议工作目录结构：

```text
/Users/palpet/.openclaw/workspace/
├── validation-lab/
│   ├── inbox/
│   ├── rounds/
│   ├── reports/
│   ├── tools/
│   ├── logs/
│   ├── state/
│   └── sandbox/
```

### 各目录用途
- `inbox/`：放主 agent 或人工发来的验证任务
- `rounds/`：每一轮验证的输入包和输出包
- `reports/`：结构化结论
- `tools/`：未来放桥接工具或脚本
- `logs/`：独立日志
- `state/`：记录 round 状态、当前版本、待验证任务
- `sandbox/`：重依赖试验区；未来 MiroFish 适合在这里做隔离试跑

---

## 7. 依赖隔离要求

未来任何 MiroFish 相关依赖都必须满足：
- 不进入主 Python 环境
- 不进入主 OpenClaw 常用 PATH
- 不修改主 agent 的运行方式
- 不污染现有启动脚本

### 推荐隔离级别
1. **目录隔离**：先分开文件和日志
2. **Python 环境隔离**：单独 venv 或 uv env
3. **服务隔离**：单独端口、单独进程、单独 stop/start
4. **容器隔离（可选）**：只有当 MiroFish 长期值得保留时才考虑 Docker

## 8. 输入协议

每次验证任务至少包含：
- 项目名
- 本轮验证目标
- 待验证假设
- 目标用户
- 当前版本材料
- 成功标准
- 失败标准

---

## 9. 输出协议

每次输出固定包含：
- 本轮验证对象
- 验证方法
- 关键发现
- 主要接受点
- 主要误解点
- 主要风险点
- 建议修正项
- 是否建议继续下一轮
- 是否建议进入真实世界验证

---

## 10. Round 机制

该 agent 应按 round 运作，而不是随意聊天推进。

### 每轮推荐结构

```text
round-001/
├── brief.md
├── audience.md
├── version-a.md
├── version-b.md
├── version-c.md
├── questions.md
├── result-summary.md
└── next-actions.md
```

### 每轮流程
#### round 输入
- 明确目标
- 提供 1–3 个版本

#### round 中间
- 比较版本
- 做验证
- 看风险和误解

#### round 输出
- 形成结论
- 指定下一轮改哪里

## 11. MiroFish 接入策略

### 当前阶段
现在不安装，只预留位置。

当前重点是：
- 设计流程
- 设计目录
- 设计协议
- 设计职责

### 真接入时顺序
1. 在 `validation-lab/sandbox/` 中做只读研究
2. 单独环境试跑最小 demo
3. 只拿一个 round 输入包做一次验证实验
4. 看结果是否真有价值，再决定是否保留

### 接入前提
只有满足以下条件，才值得正式接入：
- 不污染主系统
- 能稳定跑
- 输出对决策真有帮助
- 成本可接受
- 结论能复用到实际项目

否则宁可不用。

## 13. 与主 agent 的协作方式

- 选择要验证的命题

## 17. 执行边界摘要（可直接引用）

### validation-lab 负责什么
- 在真实投放、真实市场接触、真实用户测试之前，先做低成本验证
- 对叙事、页面、版本、市场假设进行结构化比较
- 输出风险、误解、接受点、修正建议和是否进入下一轮/真实世界验证的判断

### validation-lab 不负责什么
- 不负责主线项目执行
- 不负责主系统运行稳定性
- 不负责生产环境运维
- 不负责替代 main 做业务总调度

### 与 main 的关系
- `main` 提出要验证的命题与业务目标
- `validation-lab` 组织验证轮次并给出结论
- `main` 依据验证结果决定是否推进真实市场动作

### 当前落地原则
- 先把 agent 做成独立实验室，再考虑装工具
- 先把协议、round、目录、状态做稳定，再考虑自动化增强
- 一切未来重依赖（包括 MiroFish）默认进 `sandbox/` 或独立环境，不直接污染主 OpenClaw 系统
